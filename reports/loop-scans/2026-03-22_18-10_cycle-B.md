## Agent 1: Extended Engine (13 files scanned)

Files: analytics.py, regime.py, liquidity.py, slippage.py, fill_model.py, gap_risk.py, optimizer.py, robustness.py, signals.py, economic_calendar.py, sanity_checks.py, strategy_base.py, data_loader.py

### Confirmed Bugs — FIXED

**BUG P2: analytics.py:758-779 — after_loss_recovery always reports 100% win rate**
- `after_loss_recovery` metric only increments `loss_streak_count` and `next_win` inside the `pnl > 0` branch. Since the else branch only fires when `pnl > 0`, the `if pnl > 0` check at line 772 is a tautology — `next_win` always equals `loss_streak_count`, making win_rate = 1.0 (100%) always.
- **Fix:** Rewrote with `streak_triggered` flag. After N+ consecutive losses, the flag is set. On the next non-flat day, the outcome (win or loss) is recorded regardless. Win rate now correctly measures recovery probability.

**BUG P2: data_loader.py:252-270 — validate_bars() coverage check dead for multi-day datasets**
- When `calendar_days > 0`, `expected_bars` is computed but `coverage_pct` is never updated from its `-1.0` sentinel. Falls through to line 269: `coverage_pct = 100.0`. Multi-day datasets always pass validation regardless of missing bars.
- **Fix:** Added coverage computation and 80% threshold check inside the `calendar_days > 0` branch, matching the existing `calendar_days == 0` logic.

**BUG P3: optimizer.py:150,232 — Error trials return 0.0, biasing minimize objective**
- Optuna minimizes `-sharpe`. An error trial returning `0.0` can "beat" trials with negative Sharpe (positive `-sharpe`), corrupting `study.best_params`.
- **Fix:** Changed to `float("inf")` (worst possible for minimization) in both objective functions.

**BUG P2: structural_targets.py:103 vs eligibility_gate.py:155 — Ranging regime R:R mismatch**
- `compute_targets()` applies regime adjustment (`0.75` for ranging), allowing TP2 targets as low as 1.5R. But `evaluate_signal()` hardcodes `rr_achieved < 2.0 → SKIP`. ## Agent 3: Data + Infrastructure (~15 files scanned)

Files: src/data/loaders/*, src/data/scripts/*, src/discord/*, src/server/index.ts, scheduler.ts, db/schema.ts, middleware/*, shared/firm-config.ts

### Confirmed Bugs — FIXED

**BUG P2: duckdb-service.ts:40-44 — Timeframe SQL injection**
- `timeframe` parameter interpolated directly into S3 glob path with zero validation. `buildOhlcvQuery` validates `symbol` against allowlist but not `timeframe`. Exported function callable from multiple services.
- **Fix:** Added `ALLOWED_TIMEFRAMES` allowlist: `["1min", "5min", "15min", "30min", "1hour", "4hour", "daily"]`. Throws on invalid timeframe.

**BUG P2: duckdb-service.ts:107-115 — queryInfo() symbol SQL injection**
- `queryInfo(symbol)` interpolates `symbol` into S3 path with no validation. Called from routes with raw `req.params.symbol`.
- **Fix:** Added `ALLOWED_SYMBOLS` validation at top of `queryInfo()`.

**BUG P2: duckdb-service.ts:73-77 — AWS credentials unsanitized in SQL**
- AWS secret key interpolated into DuckDB SET statement. A key containing single quotes breaks SQL. Python counterpart (`data_loader.py:54`) already sanitizes with `.replace(chr(39), '')`.
- **Fix:** Added `sanitize()` helper that strips single quotes from all credential values.

**BUG P3: scheduler.ts:225 — preMarketPrep self-request bypasses auth**
- Fetches `localhost:4000/api/skip/today` without Authorization header. When `API_KEY` is set, auth middleware rejects the request — pre-market prep silently fails.
- **Fix:** Added `Authorization: Bearer ${process.env.API_KEY}` header when API_KEY is set.
