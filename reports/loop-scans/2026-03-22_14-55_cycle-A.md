### Confirmed Bugs — FIXED

**BUG P1: backtester.py:2564 — _compute_tier() called with wrong keyword arg**
- CLI walkforward path passes `avg_rr=wf_avg_winner_rr` but `_compute_tier()` expects `winner_loser_ratio`. Would crash with `TypeError: unexpected keyword argument 'avg_rr'` at runtime.
- Root cause: Leftover from R:R refactor in prior session — DSL and class paths were updated but CLI walkforward path was missed.
- **Fix:** Replaced with proper `winner_loser_ratio` computation from aggregated OOS trades (avg win $ / avg loss $), matching how the DSL/class paths compute it. Now passes `winner_loser_ratio=wf_winner_loser_ratio`.

**BUG P2: backtester.py:1559 — _compute_monthly_survival_stats misses zero-win months**
- `full_months = [wins for month, wins in monthly_wins.items() ...]` iterates `monthly_wins` (defaultdict). Months with 0 winning days are never incremented, so they don't appear in `.items()`. A month with 15 trading days but 0 wins is excluded from `worst_month_win_days`.
- Impact: `worst_month_win_days` is artificially inflated — never reports 0, so strategies with a catastrophic month slip through the gate.
- **Fix:** Changed to iterate `monthly_days.items()` and look up `monthly_wins.get(month, 0)`.

### Dismissed — False Positives (2)

1. **performance_gate.py:44-45** — dict access without `.get()`. FALSE POSITIVE: Early returns on lines 20-35 guard `total_trading_days < 60` and `total_trades < 100`. Stats dict always contains these keys when sourced from backtester.py. Internal function, not a public API.
2. **prop_compliance.py:482-494** — same pattern. FALSE POSITIVE: `rank_firms()` called from backtester with complete stats dict. All required keys guaranteed by caller.

- Python code from request body was interpolated into a Python script string. Single-quote escaping was insufficient — payloads with `\n` or triple quotes bypass it.

**BUG P3: agent.ts:37 — NaN propagation from query params**
- `parseInt(req.query.days)` with undefined input → NaN propagating through response.

### Confirmed Bugs — FIXED

**BUG P2: paper-execution-service.ts:391 — Silent wrong P&L with missing CONTRACT_SPECS**
- `spec?.pointValue ?? 1` fallback: if a symbol isn't in CONTRACT_SPECS, pointValue defaults to 1. For MES (pointValue=5), unrealized P&L would be 5x too low. For MNQ (pointValue=2), 2x too low.
- **Fix:** Added `if (!spec)` guard — logs warning and skips position in `updateUnrealizedPnl`. In `openPosition` (line 146), throws error immediately to prevent opening a position with unknown specs.

**BUG P2: paper-signal-service.ts:379 — Fixed UTC session windows wrong during EST**
- `SESSION_WINDOWS` used hardcoded UTC hours: NY_RTH = 13:30-20:00 UTC. This is only correct during EDT (Mar-Nov). During EST (Nov-Mar), NY RTH = 14:30-21:00 UTC. Signals during 13:30-14:30 UTC in winter would fire outside market hours.
- **Fix:** Added `isUsDst()` function (second Sunday Mar — first Sunday Nov calculation). Replaced static `SESSION_WINDOWS` with `getSessionWindows(date)` that dynamically adjusts NY RTH offset (UTC-4 EDT, UTC-5 EST). London and Asia windows remain static (not affected by US DST).

### Dismissed — False Positives (3)

1. **evolution-service.ts:262** — IS vs OOS Sharpe comparison. FALSE POSITIVE: `parentSharpe` comes from `backtests.sharpeRatio` which is the stored OOS Sharpe from the parent's walkforward run. Mutations also run walkforward. Both are OOS — not apples-to-oranges.
2. **backtest-service.ts:432** — Unbounded fire-and-forget chain. FALSE POSITIVE BY DESIGN: MC runs are intentionally non-blocking. Each chain spawns independent Python processes with their own timeouts. The backtest response returns immediately.
3. **paper-risk-gate.ts:15** — DST calculation. DISMISSED: The formula correctly computes ET offset with `getTimezoneOffset()` on a Date object.
